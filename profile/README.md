## How We Work

We do not treat technology as a side deliverable.

Our approach connects strategy, operating model, data, software, AI, and adoption into one delivery path. The goal is simple: build systems that can actually run inside the business.

Across this organization, repositories should be practical, secure, documented, and owned.

---

## Repository Expectations

Every active repository should include:

```text
README.md
.env.example
.gitignore
docs/
src/
tests/
```

A good repository README should explain:

- What the project does
- Why it exists
- Who owns it
- How to run it locally
- Required environment variables
- Architecture overview
- Testing commands
- Deployment notes
- Security notes

---

## Engineering Standards

We keep our standards direct:

- No secrets in code.
- No client data in repositories.
- Keep access limited to project need.
- Write clear documentation.
- Use pull requests for reviewed work.
- Keep branches and commits understandable.
- Build reusable components when it makes sense.
- Design for maintainability, not demos only.

---

## Branch Naming

Use clear branch names:

```text
feature/project-name-short-description
fix/project-name-issue
docs/update-readme
chore/tooling-update
```
